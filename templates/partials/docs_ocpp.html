02 OCPP — The Charger Communication Protocol OCPP 1.6j OCPP 2.0.1

OCPP (Open Charge Point Protocol) is the language chargers and central systems use to talk to each other. Every message — boot, heartbeat, authorize, meter reading — follows this protocol. Understanding it is the foundation of everything else.

OCPP Architecture — Message Flow

OpenCPO runs both protocol handlers simultaneously on separate ports. A charger connects to the port that matches its protocol version — no configuration needed on the server side.

Message Structure

Every OCPP message is a JSON array over a persistent WebSocket connection. There are three types:

// CALL — initiated by charger or central system
[2, "message-id-1234", "BootNotification", {
  "chargePointVendor": "HG Power",
  "chargePointModel": "ENC-DCL120B",
  "firmwareVersion": "2.1.4"
}]

// CALL_RESULT — response to a CALL
[3, "message-id-1234", {
  "status": "Accepted",
  "currentTime": "2026-04-02T22:00:00Z",
  "interval": 300
}]

// CALL_ERROR — error response
[4, "message-id-1234", "NotSupported", "Action not implemented", {}]

Key Messages in a Session Lifecycle

BootNotification — Charger announces itself on connect. Core responds Accepted/Pending/Rejected and sets heartbeat interval. This is where the charger is registered if new.
StatusNotification — Charger reports connector state changes: Available → Preparing → Charging → Finishing → Available. Core updates the dashboard in real time.
Authorize — Charger asks: "Can this RFID token start a session?" Core checks the token database and returns Accepted, Blocked, or Invalid. In whitelist mode, the charger can cache this response and operate offline.
StartTransaction / TransactionEvent (2.0.1) — Session begins. Core records start time, connector, token, meter reading (kWh). Returns a transaction ID used for all subsequent messages.
MeterValues — Periodic energy readings (configurable interval, typically 60–300 seconds). Used for live session tracking, billing calculation, and smart charging decisions. Includes energy (kWh), power (W), voltage, current, and SoC where supported.
StopTransaction / TransactionEvent ENDED — Session ends. Final meter reading sent, reason recorded (Local, Remote, EV Disconnect, etc.). Core calculates energy delivered, applies tariff, generates receipt.

Smart Charging — SetChargingProfile

OCPP's smart charging feature lets the central system control how much power each charger draws. This is how OpenCPO integrates with EMS to prevent grid overload:

// Core → Charger: limit to 16A (≈11kW on 3-phase)
[2, "sc-001", "SetChargingProfile", {
  "connectorId": 1,
  "csChargingProfiles": {
    "chargingProfileId": 1,
    "stackLevel": 0,
    "chargingProfilePurpose": "TxDefaultProfile",
    "chargingProfileKind": "Relative",
    "chargingSchedule": {
      "chargingRateUnit": "A",
      "chargingSchedulePeriod": [
        { "startPeriod": 0, "limit": 16.0 }
      ]
    }
  }
}]

The EMS monitors total site load and sends updated profiles when demand exceeds the grid connection capacity. Chargers respond within one OCPP message cycle (typically under 500ms).

Connecting a Charger

Configure the charger's "Central System URL" to:

# OCPP 1.6j
ws://your-server:9100/{charger_id}
wss://your-server:9100/{charger_id}   # with TLS (recommended)

# OCPP 2.0.1
ws://your-server:9201/{charger_id}
wss://your-server:9201/{charger_id}   # with TLS (mandatory in 2.0.1)

The charger_id must be unique across your network. Use a consistent naming convention: site-charger-connector (e.g., theater-amsterdam-01). The charger will auto-register on first BootNotification — no pre-configuration needed in OpenCPO.

Live Protocol Viewer — Admin → OCPP Messages shows every CALL and CALL_RESULT in real time. Invaluable for debugging charger behaviour. Filter by charger ID, message type, or time range.
