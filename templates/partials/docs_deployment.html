08 Deployment Architecture — Production-Grade Setup HA

A single server works fine for a small fleet. But EV chargers are critical infrastructure — if your central system goes down, your chargers stop working. This section covers a production architecture designed for zero-downtime deployments, geographic resilience, and zero-trust network security.

Recommended Multi-Node Deployment

Why This Architecture

Edge VPS

The only node with a public IP. It terminates TLS, applies WAF rules (CrowdSec blocks known bad actors before they reach your infrastructure), and load-balances across backend nodes. Your actual OCPP core and database are never directly exposed to the internet.

Caddy handles automatic certificate renewal via ACME — your charger WebSocket connections get proper TLS without manual cert management. Kong is an alternative for teams that need advanced rate limiting and API gateway features.

Tailscale Mesh

All inter-node communication (replication, API calls, admin access) goes over Tailscale — a WireGuard-based mesh VPN. This means your backend nodes are never accessible from the public internet, even if the edge VPS is compromised.

Tailscale's zero-trust model means you don't configure firewall rules between nodes. Nodes authenticate via device certificates, not IP allowlists. Adding a new node is a single tailscale up command.

Multi-Node HA

Three nodes give you N+2 redundancy — you can lose any two nodes and stay online. With Postgres streaming replication and Redis replication, a primary failure triggers automatic failover in <30 seconds (with Patroni or repmgr). Charger sessions continue uninterrupted because OCPP's WebSocket reconnect handles the failover transparently.

Starting Small

You don't need this architecture on day one. Start with a single VPS (4 vCPU, 8GB RAM handles ~500 concurrent chargers). Add nodes when you have the volume to justify it. The architecture is the same — you're just adding more copies of the same stack.

Data Storage — Three-Tier Model

Data Flow — Hot / Warm / Cold

The three-tier model is a pragmatic tradeoff: Redis gives sub-millisecond reads for the hot path (authorization, live status), Postgres gives durable transactional storage for business records, and TimescaleDB handles the high-volume time-series data that would bloat a standard Postgres installation. A charger sending MeterValues every 60 seconds generates ~1,440 rows per day — that's 525,000 rows per charger per year. TimescaleDB's chunk-based compression handles this efficiently.
