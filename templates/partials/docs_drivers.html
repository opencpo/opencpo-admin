04 RFID Tokens & Driver Management ISO 14443A

RFID authentication is how most EV drivers start a session today — they tap a physical card on the charger's reader, the charger sends an authorization request to OpenCPO, and charging begins or is rejected within a few hundred milliseconds. Understanding the full system — from the card's UID format to the authorization flow — prevents the most common operational headaches.

Authorization Flow — Driver Taps Card

What is an RFID Token

An RFID token is a physical NFC card — identical in form factor to a contactless bank card or hotel key. The driver holds it near the charger's NFC reader (typically the illuminated panel on the front). The reader reads the card's UID (Unique Identifier) in milliseconds and sends it to OpenCPO for authorization.

Most EV charging cards use ISO 14443A — the same standard used by Mifare Classic, Mifare Desfire, and NXP cards. The UID is either 4 bytes (single-size, most common) or 7 bytes (double-size, used in newer cards), encoded as hexadecimal: 04:A3:B7:22 or 04:A3:B7:22:C1:09:80.

Finding the UID: It's printed on the back of most charging cards. If not, use the Admin → RFID Tokens → Scan page — tap the card on any online charger and OpenCPO captures the UID from the OCPP Authorize request. No separate card reader needed.

UID Formats — Enter Them Correctly

FormatExampleNotes
Hex with colons (preferred)04:A3:B7:22Most readable, OpenCPO accepts this
Hex without separator04A3B722Also accepted — normalized on import
Hex with spaces04 A3 B7 22Accepted — normalized on import

7-byte (double)04:A3:B7:22:C1:09:80Newer cards — treat identically to 4-byte

OpenCPO normalizes all UID formats to uppercase hex without separators internally. The display format in the admin uses colons for readability. When comparing UIDs, case and separators are ignored.

Adding Tokens

Manual Entry

Admin → RFID Tokens → Add Token. Enter the UID, assign a label (driver name or fleet ID), select a token group, and set the status to Active. The token is immediately available for authorization.

Bulk Import

Upload a CSV with columns uid, label, group_id, status. OpenCPO validates all UIDs, reports duplicates, and imports the rest atomically. Use this when onboarding a fleet of 50+ drivers — preparing the CSV takes minutes, the import is instant.

# Bulk import CSV format
uid,label,group_id,status

Token Lifecycle — Status Values

StatusAuthorization resultWhen to use
ActiveAcceptedNormal in-use tokens. Driver can start sessions.
BlockedBlockedTemporarily suspended — debt, investigation, card reported lost but not yet replaced. Charger displays "Card blocked" to the driver.
ExpiredExpiredSet automatically when a token's expiry date passes. Use for contract-limited access (e.g., employee access only during employment period). Charger displays "Card expired".
InvalidInvalidUnknown UID — no record in system. This is the default for any unrecognized card tap. Displayed as "Card not recognized".

Use Blocked rather than deleting tokens — it preserves the audit trail of past sessions. Deleted tokens lose their session history association. If an employee leaves, block the token; don't delete it.

Token Groups — Fleet & Department Billing

Token Groups are how you organize drivers for consolidated billing. A group might represent a company, a department within a company, a building, or any logical unit you want to bill as one entity.

  • Each group gets a single monthly invoice covering all sessions by tokens in the group
  • Groups can have their own tariff — fleet discounts, employee rates, or free charging for a specific set of drivers
  • A token group can be suspended (all tokens blocked) without touching individual tokens — useful when a company is late on payment
  • Admin → Token Groups shows total kWh, session count, and billing amount per group for any date range

Whitelist vs Online Authorization

Online Authorization

Default mode. Every tap triggers an OCPP Authorize request to OpenCPO. Core checks the token status in real time and returns the verdict. Requires a stable network connection between charger and server. Latency: 50–200ms from tap to charger response.

Use when: You need immediate revocation capability (e.g., shared fleet where cards get lost frequently).

Local Whitelist

OpenCPO pushes the full authorized token list to the charger's local storage. The charger authorizes taps offline without contacting the server. Sessions start immediately even during a server outage or network interruption.

Use when: Network reliability is uncertain (remote sites, cellular-connected chargers) or you need sub-50ms authorization. Limitation: Revocation takes up to the next sync interval (typically 15 minutes) to propagate.

Roaming Tokens — OCPI Integration

When you set up an OCPI roaming partner (see Section 07), their drivers' tokens appear automatically in your token database under Tokens → Source: OCPI. You don't add these manually — they're pushed by the eMSP via the OCPI Tokens module.

The OCPI token flow:

  1. eMSP driver taps their Shell Recharge card on your charger
  2. Charger sends OCPP Authorize with the card UID
  3. OpenCPO checks local token database — finds it under OCPI-sourced tokens
  4. If status is ALLOWED: Accept. Session starts. OpenCPO sends a Session update and eventually a CDR back to Shell via OCPI.
  5. Shell bills their driver. You receive payment from Shell at the agreed roaming tariff.

OCPI tokens are read-only in your admin. You can see them (and why a specific tap was accepted or rejected), but you can't modify them — the eMSP owns their token data. If a roaming driver reports they can't charge, check Admin → OCPI → Token Lookup to see the token's current status as pushed by their eMSP.

Operational Best Practices

  • One card per driver, not one card per car. Cards are personal — they tie sessions to an individual for billing and audit purposes. Multiple drivers sharing one card makes billing reconciliation difficult.
  • Maintain a pool of visitor cards. Keep 5–10 unassigned active tokens for guests, contractors, and new employees before their permanent card arrives. Label them "Visitor-01" through "Visitor-10" and check them back in after use.
  • Emergency override tokens. Issue 2–3 tokens with a dedicated group flagged as "override" — these bypass tariff rules and have no expiry. Keep physical copies in a secure location for infrastructure emergencies when the main system is unreachable.
  • Set expiry dates for contractor access. When onboarding temporary staff, set a token expiry matching their contract end date. No manual cleanup needed — the token expires automatically.
  • Audit monthly. Export the token list quarterly and cross-reference with active personnel. Departed employees' cards are a common source of unauthorized usage.
03 Tariffs & Pricing

Tariffs in OpenCPO are composable — you build a tariff from individual components, then assign it to chargers or token groups. The same charger can have different tariffs for different driver groups: one rate for fleet drivers, another for the public, a third for employees charging for free.

Tariff Components

ComponentUnitExampleWhen to Use
Energy rate€/kWh€0.35/kWhAlways include this — it's the primary revenue driver
Time rate (session)€/min€0.04/minAdds predictability for slow AC chargers where session time varies widely
Idle fee€/min after grace period€0.10/min after 5 minDC chargers — discourages occupancy after session ends. Set 5–10 min grace.
Flat connection fee€/session€0.50/sessionCovers transaction costs on low-kWh sessions (e.g., top-up charging)
Minimum charge€/session€1.00 minimumProtects revenue on very short sessions

A typical public DC fast charging tariff: €0.55/kWh + €0.50 connection fee + €0.10/min idle after 5 min. This covers energy cost plus margin, recovers some overhead per session, and actively discourages charger hogging without punishing normal charging behaviour.

Fleet pricing: Fleet contracts typically run on energy-only rates with no idle fee — fleet vehicles are managed assets and occupancy is less of an issue. Consider €0.25–0.35/kWh for fleet, negotiated annually. The margin is lower but the volume is predictable.
09 The Charge App — Driver-Facing PWA

The Charge App is a Progressive Web App (PWA) — drivers add it to their phone's home screen once and it behaves like a native app, without any app store. It handles QR code scanning, OTP authentication, live session tracking, and receipt delivery.

Features

  • QR scan to charge: Each charger has a unique QR code. Driver scans it, app shows charger status and tariff, driver starts session. No account needed for one-off sessions.
  • OTP authentication: Driver enters phone number → receives SMS code → authenticated. Session tied to phone number for receipts. Faster than RFID for new users.
  • Live session tracking: Real-time power reading, kWh delivered, estimated cost, time elapsed. Data sourced from MeterValues pushed over OCPP.
  • Receipts: PDF receipt generated on session end, sent via email and available in-app. Includes session details, kWh, tariff breakdown, total cost.
  • Charger map: Integrated map showing all your chargers with live availability status. Drives organic discovery.
  • Plug & Charge provisioning: For ISO 15118-capable setups — driver requests a contract certificate through the app, which OpenCPO issues and delivers to the vehicle.

Customization

The Charge App uses a skins system — you can brand it with your own logo, colors, and name without forking the codebase. The skin configuration lives in the OpenCPO settings and is served at build time. Multiple brands can share the same backend. Each roaming partner can have their own skinned instance pointing at the same charger network.

10 Virtual Charger Farm — Load Testing & Compliance

The Charger Farm is a simulator that creates virtual chargers and runs realistic charging scenarios against your OpenCPO instance. It's how you verify your setup works before deploying physical hardware — and how you find out where your system breaks under load.

What It Simulates

  • BootNotification and heartbeat cycles — validates charger registration and heartbeat interval handling
  • Full session lifecycle — Authorize → StartTransaction → MeterValues stream → StopTransaction, with randomized energy and duration
  • Mixed OCPP versions — simultaneous 1.6j and 2.0.1 chargers in the same test run
  • Network degradation — simulated packet loss, reconnects, and offline periods to test OCPP reconnect handling
  • Peak load scenarios — 100, 500, 1000 concurrent sessions to establish your throughput ceiling
  • Chaos mode — random disconnects, malformed messages, unexpected message ordering
  • Hardware profiles — simulates specific charger models (HG Power ENC-DCL120B, ABB Terra series) with their known quirks

When to Use It

  1. Before go-live: Run a 24-hour soak test with 50 virtual chargers to catch any edge cases in your tariff engine, session handling, or database queries under sustained load.
  2. After a software update: Run the compliance scenario set to verify OCPP message handling is correct and no regressions were introduced.
  3. Capacity planning: Establish the charger count your current server handles before performance degrades — then size your infrastructure ahead of that limit.
  4. Debugging: Reproduce a specific charger's behaviour by configuring a Farm charger with the same firmware version and sequence of messages.
11 API Reference REST

OpenCPO exposes a versioned REST API for all management operations. Everything the admin panel does, the API can do — enabling integration with your own systems, automated provisioning, and external dashboards.

Authentication

# Management API — X-API-Key header
curl https://api.your-opencpo.com/api/v1/chargers \
  -H "X-API-Key: your-management-api-key"

# OCPI endpoints — Bearer token (per-partner)
curl https://api.your-opencpo.com/ocpi/2.2.1/locations \
  -H "Authorization: Token ocpi-partner-token-here"
