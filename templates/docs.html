01 Getting Started

OpenCPO is an open-source OCPP central system with a built-in Certificate Authority, OCPI roaming, and a full management stack. You own your infrastructure, your data, and your PKI — no per-charger fees, no vendor dependency, no hosted cloud you can't audit.

What is a CPO?

A Charge Point Operator manages physical EV chargers and the infrastructure behind them. As a CPO, you own (or manage) the chargers, connect them to your central system, set pricing, manage driver access, and handle billing. The energy sold through your chargers is your revenue. OpenCPO is the central system.

What OpenCPO Handles

OCPP 1.6j and 2.0.1 protocol handling — chargers connect and communicate over WebSocket. Authorization (RFID, OTP, certificate-based). Tariff calculation and billing. PKI and certificate issuance. OCPI roaming to external networks. Admin dashboard. Driver-facing PWA.

Quick Install

# 1. Clone
git clone https://github.com/opencpo/opencpo.git
cd opencpo

# 2. Install Python dependencies
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 3. Configure
cp .env.example .env
# Edit .env: set DATABASE_URL, REDIS_URL, SECRET_KEY, MANAGEMENT_API_KEY

# 4. Initialize database (runs Alembic migrations)
alembic upgrade head

# 5. Start OCPP Core (handles charger connections)
python ocpp_core/main.py &

# 6. Start Admin panel
python admin/main.py

First Login

The admin panel is available at http://localhost:8080 by default. On first visit, you'll be guided through a 6-step setup wizard to create an admin account, configure your organization, and set up SMTP — no pre-seeded credentials needed.

First Charger — Minimal Setup

  1. Configure your charger's "Central System URL": ws://your-server:9100/{charger_id} for OCPP 1.6j, or ws://your-server:9201/{charger_id} for OCPP 2.0.1. Replace {charger_id} with a unique identifier (e.g., charger-001).
  2. Power cycle or reconnect the charger. It will send a BootNotification — OpenCPO auto-registers it.
  3. Open Admin → Chargers. Your charger appears with status Available.
  4. Create a tariff (Admin → Tariffs) and assign it to the charger.
  5. Add an RFID token (Admin → RFID Tokens) and tap it on the charger. Session starts.
TLS from day one: For production, put a reverse proxy (Caddy) in front and use wss:// instead of ws://. Caddy with automatic ACME makes this a single configuration block. Chargers on plain WebSocket are unencrypted — RFID UIDs and session data are visible on the network.

12b Monitoring & Troubleshooting

Most charger issues resolve themselves — a momentary network dropout, a charger rebooting after a firmware update, a driver tapping too quickly. What you need is visibility into the ones that don't self-resolve, and a fast path to root cause when they matter.

Common Issues & Diagnosis

SymptomFirst checksDiagnostic tool
Charger shows Offline Ping the charger IP. Check the network path between charger and server. Verify the WebSocket URL configured on the charger exactly matches ws[s]://host:port/{charger_id}. Check for firewall rules blocking the OCPP port. Admin → OCPP Messages — look for the last message and timestamp
Session won't start (RFID tap) Check the RFID token status in Admin → Tokens. Verify it is Active, not Blocked or Expired. Check the OCPP Messages log for the Authorize request — it shows the exact response code returned. Admin → OCPP Messages → filter by charger_id and message type "Authorize"
OCPI partner can't reach your chargers Verify the OCPI credentials were exchanged successfully (Admin → OCPI Roaming → partner status should show Active). Check that your Locations module is returning valid data. Verify your server is reachable from the partner's IP. Admin → OCPI Roaming → Log — shows every HTTPS request and response
Certificate rejected at TLS handshake Check certificate expiry on the PKI page. Verify the full certificate chain is installed on the charger (leaf cert + sub-CA cert, not just the leaf). Check that the charger trusts your Root CA. Admin → PKI → certificate detail → verify chain validity
Sessions stopping unexpectedly Check the StopTransaction reason in the session detail — "EVDisconnected" is normal, "Other" or "DeAuthorized" indicates an issue. Check for power outages on site (MeterValues will show zero power readings). Admin → Sessions → session detail → OCPP message trace

Live Diagnostic Tools

  • Admin → OCPP Messages — live protocol viewer. Every CALL and CALL_RESULT in real time. Filter by charger, message type, or time range. The single most useful tool for debugging charger behaviour.
  • Admin → OCPI Roaming → Log — HTTP request/response log for every OCPI API call. Includes request headers, body, and response status. Essential for debugging roaming integration issues.
  • Admin → Dashboard — real-time charger status overview. Color-coded by status: Available (green), Charging (blue), Faulted (red), Offline (grey).
